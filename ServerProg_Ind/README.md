# Индивидуальное задание – Бэкенд для аукционов

В этом задании вы самостоятельно спроектируете и реализуете серверную часть **Lily Market** — кампусного маркетплейса, где верифицированные студенты могут выставлять товары на аукцион, делать ставки в реальном времени и получать живые обновления прямо на телефоне. В качестве ориентира вам предоставлен готовый фронтенд.

[Lily Market](https://xivol.github.io/greenswamp/indiv/auction/auctions)

**Lily Market** — это **платформа с приоритетом мобильных устройств**. Ответы API должны быть лаконичными, события реального времени — понятными на маленьком экране, а задержка важнее, чем в десктопных приложениях.

Тестирование и документирование является необходимой частью разработки. 

---

## Правила аукциона

Сервер является **единственным источником истины** в отношении состояния аукциона и проверки ставок. 

### Жизненный цикл аукциона

Аукцион проходит через следующие состояния:

| Состояние | Значение |
|---|---|
| `Active` | Открыт для ставок |
| `Ended` | Время истекло, победитель определён |
| `Sold` | Победитель подтвердил сделку |
| `Canceled` | Продавец отменил до первой ставки |

### Правила ставок

- Делать ставки могут только аутентифицированные студенты.
- Продавец не может делать ставки на собственный аукцион.
- Ставка должна превышать текущую наивысшую ставку не менее чем на `minimumIncrement`.
- Ставки, поданные после `endTime`, должны отклоняться — сравнение всегда идёт с серверным временем, а не клиентским.
- Если задана `buyNowPrice` и входящая ставка достигает или превышает её, аукцион закрывается немедленно и этот участник побеждает.
- Первая ставка фиксирует начальную цену. Аукцион, на который уже есть хотя бы одна ставка, продавец отменить не может.

### Окончание аукциона

Победитель определяется когда:

- Наступает `endTime` аукциона — побеждает участник с наивысшей валидной ставкой, **либо**
- Ставка достигает `buyNowPrice` — этот участник побеждает немедленно.

Если аукцион завершился без ставок, победитель не назначается и продавец получает уведомление.

## Что нужно реализовать

Ваш бэкенд должен поддерживать как минимум следующее:

- **Регистрацию.** Пользователи регистрируются по университетской почте.
- **Создание аукционов и управление ими.** Продавцы создают аукционы с названием, начальной ставкой, шагом, временем окончания и необязательной ценой «купить сейчас». Продавцы могут редактировать или отменять свои аукционы до первой ставки.
- **Ставки.** Покупатели подают ставки. Сервер проверяет их, сохраняет и транслирует обновление в реальном времени.
- **Завершение аукциона.** Аукционы закрываются автоматически по `endTime` согласно серверному времени. Фоновый процесс находит истёкшие аукционы, определяет победителя и уведомляет обе стороны.
- **Уведомления.** Пользователи получают уведомления о: принятой ставке, перебитой ставке, скором завершении аукциона (за 5 минут), победе, завершении без ставок.

Всё остальное — на ваше усмотрение.

## Архитектурное решение: REST, SignalR или гибрид

Это главный вопрос проектирования в задании. Вы обязаны ответить на него явно в README.

Три реалистичных варианта:

**А — Чистый SignalR.** Вся коммуникация клиента с сервером проходит через Hub. Ставки передаются вызовами методов Hub, обновления — Hub-событиями. Низкие накладные расходы, низкая задержка, ориентировано на постоянное соединение. Хорошо работает на мобильных устройствах при стабильном подключении.

**Б — Гибридный подход.** Команды (ставка, создание аукциона, отмена) — через REST-эндпоинты. События (обновление ставки, уведомление о перебитой ставке, завершение аукциона, предупреждение о скором закрытии) — через SignalR. REST даёт стандартную HTTP-семантику и упрощает тестирование; SignalR берёт на себя широковещательный слой.

**В — Чистый REST с polling.** Постоянного соединения нет, клиент периодически опрашивает сервер. Просто реализовать, плохо подходит для торгов в реальном времени и особенно невыгодно на мобильных устройствах из-за расхода батареи и трафика — допустимо только при наличии убедительного обоснования в README с честным разбором недостатков.

Единственно верного ответа нет. Хорошо обоснованный и корректно реализованный вариант Б получит ту же оценку, что и хорошо обоснованный и корректно реализованный вариант А. Оценка снижается, если выбор сделан без объяснения причин или если реализация не соответствует заявленному подходу.

В README должен быть раздел **«Архитектурное решение»** (один-два абзаца), в котором указано: какой вариант выбран, почему, от чего вы отказались, не выбрав альтернативы, и был ли этот компромисс оправдан для данной задачи.

## Требования к тестам

Тесты должны запускаться командой `dotnet test` без дополнительной настройки.

Обязательно протестировать:

- **Валидацию ставок** — корректные ставки принимаются, некорректные отклоняются. Покройте как минимум: обычную валидную ставку, ставку ниже минимального шага, ставку на завершённый аукцион, ставку продавца на собственный аукцион, срабатывание цены «купить сейчас».
- **Логику завершения аукциона** — корректный выбор победителя по истечении времени, корректная обработка аукциона без ставок, немедленное закрытие по цене «купить сейчас».
- **Асинхронность** — покажите, что две ставки, поданные одновременно на один аукцион, не приводят к повреждению состояния (например, оба не могут быть приняты как победные, а текущая ставка не может оказаться некорректной).

Количество тестов и выбор фреймворка — на ваше усмотрение. Важно, чтобы тесты были содержательными и охватывали перечисленные области.

## Нефункциональные требования

- **Целевой фреймворк:** .NET 8-10
- **База данных:** PostgreSQL или SQL Server через EF Core. Данные аукциона должны сохраняться — перезапуск сервера не должен уничтожать активные аукционы или историю ставок.
- **Аутентификация:** на основе JWT. Эндпоинты создания аукционов и подачи ставок должны требовать валидный токен.
- **Дизайн API с приоритетом мобильных устройств:** держите ответы компактными, пагинируйте списки аукционов, не возвращайте поля, которые мобильному клиенту не нужны.
- **CORS:** настройте так, чтобы предоставленный фронтенд и ваш тестовый клиент могли подключаться в режиме разработки.
- **Логирование:** логов должно быть достаточно, чтобы по ним можно было проследить полный жизненный цикл аукциона — создание, каждую ставку, завершение, выбор победителя — без чтения кода.

## Советы

**Сначала модель, потом проводка.** Добейтесь, чтобы валидатор ставок, машина состояний аукциона и логика завершения работали и были покрыты тестами, прежде чем касаться SignalR или HTTP. Это чистые C#-классы без инфраструктурных зависимостей — их быстро написать и быстро проверить.

**Продумайте стратегию конкурентности заранее.** Несколько мобильных клиентов могут подать ставки в одну миллисекунду. Решите сразу, как будете защищать общее состояние — локом на аукцион, `SemaphoreSlim` или транзакцией базы данных с блокировкой строк — и применяйте это последовательно. Тест на конкурентность вскроет любые пробелы.

**Переподключение нужно планировать заранее.** Обрыв и восстановление соединения на мобильном устройстве — это штатная ситуация, а не граничный случай. Подумайте, какой информации достаточно, чтобы привязать новое SignalR-соединение к текущему аукциону и отправить клиенту актуальное состояние.

**В гибридном подходе есть проблема идентификации.** REST-запрос не несёт встроенной идентичности, как соединение SignalR. JWT решает эту проблему на стороне REST; для SignalR убедитесь, что можете идентифицировать пользователя из контекста соединения. Не стройте две отдельные системы идентификации.

**Завершение аукциона — забота фонового сервиса, а не запроса.** Используйте `BackgroundService` с регулярным интервалом. Не полагайтесь на клиента для запуска завершения и не решайте состояние аукциона внутри метода SignalR Hub. Серверные часы — авторитетны; клиентский обратный отсчёт — декоративен.

**Документируйте по ходу работы.** Документация API — лёгкий пункт, если писать её параллельно с кодом. Если оставить на последний вечер перед сдачей, превратится в каторгу.
